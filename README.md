# Azure DevOps Team Productivity Analytics

Инструмент для автоматического сбора и анализа метрик продуктивности команды из Azure DevOps. Генерирует подробные HTML и Excel отчёты с интерактивными графиками.

## Описание

Инструмент собирает следующие данные из Azure DevOps:
- **Коммиты** — количество, добавленные/удалённые строки, активность по авторам
- **Pull Requests** — создание, слияния, статусы
- **Ревью** — голоса рецензентов, комментарии в PR
- **Рабочие элементы** — созданные, закрытые и назначенные задачи/баги/истории

## Настройка

### Шаг 1: Создайте PAT в Azure DevOps

**Для Azure DevOps Services (dev.azure.com):**

1. Войдите на [dev.azure.com](https://dev.azure.com) в свою организацию
2. Нажмите на иконку пользователя (верхний правый угол) → **Personal access tokens**
3. Нажмите **+ New Token**
4. Заполните форму:
   - **Name**: `TeamAnalytics` (или любое название)
   - **Organization**: выберите нужную организацию
   - **Expiration**: установите срок действия (например, 90 дней)
   - **Scopes**: выберите **Custom defined** и отметьте:
     - **Code**: `Read`
     - **Work Items**: `Read`
5. Нажмите **Create** и **скопируйте токен** (он отображается только один раз!)

**Для Azure DevOps Server (on-premises):**

1. Перейдите в веб-интерфейс ADO Server: `https://your-server/tfs`
2. Нажмите на иконку пользователя → **Security**
3. Раздел **Personal access tokens** → **Add**
4. Заполните поля аналогично облачному варианту

### Шаг 2: Создайте файл .env

```bash
# Скопируйте пример конфигурации
cp .env.example .env
```

Откройте файл `.env` в текстовом редакторе и заполните:

```env
# Для Azure DevOps Services (облако)
ADO_SERVER_URL=https://dev.azure.com/your-organization
ADO_PAT=your-pat-token-here

# Для Azure DevOps Server (on-premises)
# ADO_SERVER_URL=https://tfs.company.local/tfs
# ADO_PAT=your-pat-token-here

# Проекты (через запятую)
ADO_PROJECTS=MyProject,AnotherProject

# Участники команды (через запятую, необязательно)
# Если пусто — будут собраны данные по всем авторам
ADO_TEAM_MEMBERS=ivan@company.ru,maria@company.ru

# Период анализа
DATE_FROM=2024-01-01
DATE_TO=2024-03-31
```

## Использование

### Базовый запуск

```bash
# Использует настройки из .env
python main.py
```

### Параметры командной строки

```bash
python main.py [--from ДАТА] [--to ДАТА] [--format ФОРМАТ] [--output ДИРЕКТОРИЯ] [--config ФАЙЛ]
```

| Параметр | Описание | По умолчанию |
|---|---|---|
| `--from` | Начало периода (YYYY-MM-DD) | из .env |
| `--to` | Конец периода (YYYY-MM-DD) | из .env |
| `--format` | Формат: `html`, `excel`, `both` | `both` |
| `--output` | Директория для отчётов | `./output` |
| `--config` | Путь к .env файлу | `.env` |

### Примеры команд

```bash
# Отчёт за первый квартал 2024
python main.py --from 2024-01-01 --to 2024-03-31

# Только HTML за февраль
python main.py --from 2024-02-01 --to 2024-02-29 --format html

# Excel в указанную папку
python main.py --from 2024-01-01 --to 2024-06-30 --format excel --output C:\Reports

# Использовать другой .env файл
python main.py --config /path/to/project.env

# Переопределить дату из командной строки (игнорирует .env)
python main.py --from 2024-03-01 --to 2024-03-31 --format html
```

## Описание метрик

Все метрики считаются **по конкретному участнику** за выбранный период. Ниже описано, что именно учитывается в каждом показателе.

### Коммиты

| Метрика | Как считается |
|---|---|
| **Коммиты** | Количество коммитов, у которых поле `author.email` / `author.name` совпадает с участником. Мерж-коммиты учитываются наравне с обычными. |
| **+ строк** | Сумма изменений типа `Add` и `Edit` во всех файлах коммита (по полю `change_counts` из API). Это количество затронутых файловых операций, а не буквально строк кода. |
| **− строк** | Количество изменений типа `Delete` в файлах коммита (удалённые файлы). |

### Pull Requests

| Метрика | Как считается |
|---|---|
| **PR создано** | Количество PR, у которых поле `created_by` совпадает с участником, а дата создания попадает в период. |
| **PR влито** | Подмножество созданных участником PR со статусом `Completed` и результатом слияния `Succeeded`. |
| **PR закрыто** | Подмножество созданных участником PR со статусом `Abandoned` (закрыты без слияния). |

> **Важно:** метрики PR (создано / влито / закрыто) учитывают только те PR, которые участник **создал сам**. PR, в которых он был рецензентом, здесь не отражаются.

### Ревью

| Метрика | Как считается |
|---|---|
| **Ревью** | Количество PR, в которых участник добавлен как рецензент (независимо от голоса). Каждый PR считается один раз, даже если участник пересматривал его несколько раз. |
| **Одобрено** | Из всех ревью — те, где участник проголосовал «Approved» (vote = 10). |
| **С замечаниями** | Голоса «Approved with suggestions» (vote = 5). |
| **Отклонено** | Голоса «Rejected» (vote = −10). |
| **Комментарии** | Суммарное количество отдельных сообщений, написанных участником в тредах PR (системные сообщения ADO исключаются). Учитываются комментарии во **всех** PR репозиториев, входящих в заданные проекты, — не только в тех, которые участник создал или проревьюировал голосом. |

> **Пример:** если в одном PR участник написал 2 комментария в разных тредах и ещё 3 в одном треде — в метрику войдёт 5.

### Рабочие элементы (Work Items)

| Метрика | Как считается |
|---|---|
| **WI создано** | Рабочие элементы, у которых поле `CreatedBy` совпадает с участником, а `CreatedDate` попадает в период. |
| **WI закрыто** | Рабочие элементы в состоянии `Closed`, `Resolved` или `Done`, у которых `ClosedDate` попадает в период. Атрибутируются полю `AssignedTo` (или `CreatedBy`, если назначение не задано). |
| **WI назначено** | Рабочие элементы, у которых `AssignedTo` совпадает с участником, а `ChangedDate` попадает в период (т. е. задачи, которые **сейчас назначены** на участника и изменялись в периоде). |

> **Примечание:** один и тот же рабочий элемент может попасть сразу в несколько категорий. Например, задача, созданная и закрытая участником в рамках периода, увеличит и «WI создано», и «WI закрыто».

## Формат выходных файлов

### HTML-отчёт (`report_YYYYMMDD_YYYYMMDD.html`)

Самодостаточный HTML-файл с интерактивными графиками Plotly:
- **Сводная таблица** — все метрики по каждому участнику
- **График коммитов** — коммиты по неделям (столбчатый, с разбивкой по авторам)
- **График PR** — созданные и влитые PR по неделям
- **График ревью** — ревью по неделям
- **График рабочих элементов** — созданные и закрытые задачи
- **Тепловая карта** — суммарная активность (неделя × участник)

Файл можно открыть в любом браузере без подключения к интернету (кроме загрузки Plotly.js через CDN).

### Excel-отчёт (`report_YYYYMMDD_YYYYMMDD.xlsx`)

Многостраничный Excel-файл:
- **Информация** — метаданные отчёта (период, проекты, количество участников)
- **Сводка** — все метрики по каждому участнику
- **По неделям** — данные с разбивкой по неделям
- **[Имя участника]** — отдельный лист для каждого участника (если < 20 человек)

## Устранение проблем

### Ошибка аутентификации (401)
- Проверьте правильность PAT в `.env`
- Убедитесь, что токен не истёк
- Проверьте права токена (Code: Read, Work Items: Read)

### Ошибка подключения
- Проверьте `ADO_SERVER_URL` — должен быть доступен с вашего компьютера
- Для on-premises проверьте настройки VPN/прокси

### Нет данных в отчёте
- Убедитесь, что в `ADO_PROJECTS` указаны реальные имена проектов
- Проверьте период (`DATE_FROM`, `DATE_TO`)
- Если указаны `ADO_TEAM_MEMBERS` — убедитесь, что написание совпадает с ADO

### Ошибка при установке пакетов
```bash
# Обновите pip
python -m pip install --upgrade pip

# Установите пакеты заново
pip install -r requirements.txt --force-reinstall
```
